## Текущее состояние проекта

IFF уже не только идея.
В коде есть отдельный IFF-слой:

- кнопка `IFF` на основном PDA;
- отдельная форма `IffActivity`;
- вкладки `КОНТАКТ`, `КОМАНДА`, `КАРТА`;
- локальный roster;
- `Я ПОДХОЖУ`;
- `TRUST` / `UNTRUST`;
- BLE witness;
- confidence layers;
- combat state;
- диагностическая запись `ЗАПИСАТЬ`.

На двух физических телефонах уже проверен BLE-жизненный цикл:

```text
CURRENT_SINGLE -> STALE -> UNKNOWN
```

Это подтверждает, что phone-to-phone BLE может быть основой для текущего контакта.
Но это еще не точные координаты, не криптографическая идентичность и не надежное направление.

## Завтрашний офисный ресурс

Завтра в офисе будет 3 телефона.

Плановая роль телефонов:

```text
Телефон A - неподвижный свидетель 1.
Телефон B - неподвижный свидетель 2.
Телефон C - идущий игрок / Петя / "я подхожу".
```

Один из неподвижных телефонов желательно подключить к Codex/ADB, чтобы можно было смотреть его состояние.
На остальных можно собирать логи.

В офисе 10-20 метров может быть сложно.
Рабочий диапазон теста:

```text
7-14 м между свидетелями - достаточно для офисного первичного теста.
```

## Главная договоренность

Нам нужен не только режим `я слышу Петю`.
Нам нужен обмен witness-данными между свидетелями, чтобы строить более полезную оценку:

```text
кто слышит Петю;
насколько свежий сигнал;
какой RSSI;
где находятся свидетели;
как меняется сигнал во времени;
где вероятная область Пети;
какое примерное направление и расстояние от наблюдателя.
```

Без обмена каждый телефон знает только свою локальную картину.
Для более точного направления и расстояния во время игры нужен общий набор наблюдений.

## Ограничение группами

Обмен witness-данными нужно ограничивать, иначе можно перегрузить радио, батарею, UI и основную игровую логику.

Рабочая модель:

```text
Команда = все свои в roster.
IFF-группа = те, кто участвует в локальной тактической оценке.
Контакт = конкретный подходящий игрок.
```

Ограничения:

- обрабатываем только известных своих;
- в будущем фильтруем по группе;
- усиленный обмен включается вокруг активного `я подхожу`;
- witness-report живет коротко и устаревает;
- не считаем старые сигналы текущим контактом;
- в расчете учитываем несколько лучших свежих свидетелей, а не всех подряд.

Практическая MVP-идея:

```text
1 свидетель - базовое подтверждение.
2 свидетеля - полезная оценка области/направления.
3 свидетеля - лучше для устойчивости.
4+ свидетеля - прирост меньше, можно показывать как "еще N слышат".
```

## Что передавать между свидетелями

Не передавать карту, историю и весь UI.
Передавать короткие witness-report:

```text
witnessId
targetPlayerId
RSSI
ageMs
source = BLE / Wi-Fi / other
witness lat/lon
witness accuracyMeters
timestamp
trust/known status
```

Идущий игрок, если GPS доступен, должен передавать:

```text
playerId
approachState
lat/lon
accuracyMeters
locationAgeMs
heading/speed, если есть
timestamp
radio token / claim
```

## Роль GPS

GPS нельзя игнорировать.
Если в лесу GPS работает нормально, это один из основных рабочих слоев.

Правильная модель:

```text
GPS отвечает за "где".
BLE/RSSI отвечает за "сейчас рядом".
Свидетели отвечают за "не только мой телефон это видит".
```

Если GPS Пети хороший:

```text
Петя передает координаты.
Телефон наблюдателя считает дистанцию и азимут.
Компас/heading переводит это в "11 часов".
BLE/witness подтверждают свежую близость.
```

Пример UI:

```text
ПЕТЯ ПОДХОДИТ
11 часов
расчет 18 м
зона 10-25 м
GPS Пети: 6 м / 2 сек
радио: свежий сигнал
свидетели: Вася, Женя
```

## Плохой GPS

Если GPS Пети плохой, не рисуем точную точку.
Используем свидетелей как опорные точки.

Расчет строится по:

- текущей силе сигнала у свидетелей;
- изменению RSSI во времени;
- позициям свидетелей;
- свежести данных;
- качеству GPS самих свидетелей.

Пример:

```text
Вася слышит Петю сильно.
Женя слышит Петю слабее.
Значит Петя вероятнее ближе к Васе.

У Васи RSSI растет.
У Жени RSSI падает.
Значит Петя движется к Васе или в сторону Васи.
```

При плохом GPS показывать:

```text
расчетную область;
сектор;
зону доверия;
направление не подтверждено, если данных мало.
```

Не показывать ложную точную точку.

Рабочие источники позиции:

```text
SELF_GPS
WITNESS_RADIO_GEOMETRY
MIXED_GPS_RADIO
UNKNOWN
```

## Wi-Fi Direct

Wi-Fi Direct не нужен как первый завтрашний тест.

Он может быть полезен позже как транспорт для обмена witness-report, но:

- это тяжелее BLE;
- нужно писать discovery/connect/group-owner/socket-логику;
- нужны Android-разрешения;
- может потребоваться пользовательское подтверждение соединения;
- это может съесть офисное время.

Для ближайшего теста лучше:

```text
BLE witness + логи на трех телефонах.
```

Полноценную сеть между телефонами делать позже, когда станет понятно, какие witness-report реально нужны.

## Завтрашние тесты

Сценарии:

```text
A и B лежат на расстоянии 7 м.
C подходит к A.
C стоит между A и B.
C подходит к B.
C в кармане.
C за телом.
C уходит / выключает передачу.

Повторить по возможности при расстоянии A-B 10-14 м.
```

Что хотим увидеть:

```text
A слышит C сильнее -> C ближе к A.
B слышит C сильнее -> C ближе к B.
A и B слышат похоже -> C примерно между ними.
RSSI растет у A -> C приближается к A.
RSSI устарел -> не считать текущим контактом.
```

Цель теста:

```text
не получить идеальные координаты;
а проверить, можно ли построить честную оценку области, направления и близости по свидетелям.
```
