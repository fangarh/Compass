# 2026-05-18 Радиодетекция Wi-Fi BLE

## Решение

Сохранить Wi-Fi как необходимое зло для текущей версии радиодетекции.

После обсуждения с заказчиком принято: сейчас нет возможности перейти на BLE. BLE остается технически интересным направлением, но не является ближайшим планом внедрения, пока не изменятся ограничения проекта и оборудования.

## Wi-Fi

Роль Wi-Fi:

- дальнее обнаружение объекта;
- совместимость с существующими игровыми модулями;
- fallback для старых сценариев;
- диагностические режимы для тестов.

Текущий статус: Wi-Fi использовать вынужденно, потому что заказчик уже опирается на Wi-Fi-чипы/маяки и переход на BLE пока недоступен организационно или технически.

Ограничение: современные Android ограничивают частый `WifiManager.startScan()`.
Даже если приложение просит сканы чаще, система может возвращать cached results.

Рабочий критерий свежего Wi-Fi scan:

```text
WIFI_DIAG event=results source=receiver updated=true
```

## BLE

Статус BLE: отложено.

Потенциальная роль BLE в будущем:

- более частые proximity-обновления рядом с объектом;
- меньшее энергопотребление;
- уточнение ближней зоны после Wi-Fi обнаружения;
- перспективная замена короткой радиодетекции.

На текущем этапе BLE не планируется как замена Wi-Fi, пока заказчик не подтвердит возможность перехода оборудования и сценариев.

Стартовые зоны по RSSI:

```text
RSSI > -60 dBm     очень близко
-60..-75 dBm       близко/средне
-75..-85 dBm       далеко/слабо
ниже -85 dBm       нестабильный край
нет пакетов 10-15 сек - потеря
```

## Целевая полевая схема

```text
Wi-Fi: 1 scan каждые 35-60 секунд
BLE: continuous balanced scan
BLE low-latency: короткие bursts во время активного поиска
```

## Единая идентичность объекта

Wi-Fi и BLE не должны создавать разные игровые объекты.
У физического объекта должен быть один `objectId` и несколько radio identifiers.

Пример:

```text
Wi-Fi BSSID AA:BB:CC:11:22:33 -> anomaly_042
BLE UUID/Major/Minor 1/42     -> anomaly_042
```

Дистанцию не нужно сливать в одно значение в метрах. Лучше переводить каждый источник в игровые зоны и затем комбинировать зоны.

## Следующие шаги

- Протестировать текущий 5-секундный Wi-Fi debug build в поле.
- Получить от заказчика Wi-Fi-чип/модуль, который они используют.
- После получения провести анализ чипа: режим работы, SSID/BSSID, частота/каналы, мощность, beacon interval, питание, стабильность обнаружения Android-телефонами.
- На основе анализа уточнить стратегию Wi-Fi-детекции и требования к приложению.
- BLE-маяки и BLE scanning держать в backlog, но не планировать как ближайший переход.
- Unified object mapping сохранять как архитектурную идею на будущее, если появится несколько radio identifiers для одного игрового объекта.
